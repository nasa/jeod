100 /**
101  * This class provides the interface between the macros in jeod_alloc.hh and
102  * the rest of the JEOD memory model.
103  * The public interface is via the publicly visible static methods.
104  * All nonstatic member functions are private.
105  * Each public static method relays the method call to the singleton memory
106  * manager via a correspondingly named private member function.
107  *
108  * \par Singleton
109  * The class is intended to be a singleton.
110  * The private static member JeodMemoryManager::Master points to this
111  * singular instance. The constructor sets that static member if it is null.
112  * The constructor issues a non-fatal error when multiple instances of the
113  * class are created.
114  *
115  * \par Thread Safety
116  * This class contains objects that must be accessed and updated in a
117  * thread-safe manner. The member data that must be used atomically are
118  * - JeodMemoryManager::alloc_table - Maps memory addresses to memory items
119  * - JeodMemoryManager::type_table - Maps RTTI names to type descriptors
120  * - JeodMemoryManager::string_table - Maps unique strings to themselves.
121  * - JeodMemoryManager::cur_data_size - Current size of allocated data.
122  * - JeodMemoryManager::max_data_size - Maximum of the above.
123  * - JeodMemoryManager::max_table_size - Maximum allocation table size.
124  * - JeodMemoryManager::allocation_number - Number of allocations made.
125  * \par
126  * To ensure the constraint is satisfied, access to the these elements is
127  * protected by means of a mutex and is limited to a small number of methods.
128  * A pair of methods, JeodMemoryManager::begin_atomic_block and
129  * JeodMemoryManager::end_atomic_block systematize the use of the mutex.
130  * The methods that operate on the protected data are
131  * - Constructor and destructor.@n
132  * The constructor operates on the protected data before it creates the
133  * mutex and marks the JeodMemoryManager object as usable. The destructor
134  * marks the object as unusable and destroys the mutex before operating
135  * on the protected data.
136  * - JeodMemoryManager::generate_shutdown_report, which is called by the
137  * destructor after it has destroyed the mutex.
138  * - Methods whose names end with _atomic. These methods use the
139  * begin_atomic_block / end_atomic_block paradigm to ensure that the
140  * operations are carried out atomically.
141  * - Methods whose names end with _nolock. These methods operate on protected
142  * data but do so without atomic protection. These methods are called
143  * only by _atomic methods from within their atomic protection block.
144  *
145  * \par Forbidden Word - Mutable
146  * The data member JeodMemoryManager::mutex is mutable, a forbidden word
147  * per the JEOD coding standards. The coding standards allow for waivers to
148  * the standards if the exception is justified. This section provides the
149  * explanation needed to enable the use of that word in this case.
150  * \par
151  * The @a mutable keyword tells the compiler to ignore modifications to
152  * mutable elements in an otherwise @a const method. The @a mutex is mutable
153  * because, athough its value does change with a successful lock, it is restored
154  * to its prelock value with an unlock. A method that could otherwise qualify
155  * as a const method can still be a const method by marking the mutex as
156  * mutable. Mutexes are one of the well-accepted types of data that typically
157  * marked as mutable.
158  *
159  * \par Assumptions and Constraints on the Simulation Developer
160  * This class places restrictions on the simulation developer.
161  * - The simulation's MessageHandler object must be constructed prior
162  * to constructing the simulation's JeodMemoryManager object.
163  * - The simulation's MessageHandler object must not be destroyed prior
164  * to constructing the simulation's JeodMemoryManager object.
165  * - The simulation's JeodMemoryManager object must be constructed prior to
166  * invoking any of the JEOD_ALLOC_xxx macros in other models.
167  * - The simulation's JeodMemoryManager object must not be destroyed before
168  * other models release their allocated memory.
169  * \par
170  * The recommended solution is to create an instance of a compliant
171  * SimulationInterface before creating any other models and to destroy
172  * that SimulationInterface object after destroying all other models.
173  * A simple way to achieve this in a Trick-07 simulation is to define a
174  * Trick sim object that contains a TrickSimulationInterface element and
175  * to place this sim object immediately after the sys sim object.
176  *
177  * \par Assumptions and Constraints on the Simulation Engine
178  * This class makes certain assumptions of the behavior of the simulation
179  * engine.
180  * - The simulation engine will not spawn threads that use the JEOD memory
181  * model to allocate memory until after the SimulationInterface object has
182  * been constructed.
183  * - The simulation engine will join all threads that use the JEOD memory
184  * model prior to destroying the SimulationInterface object.
185  * \par
186  * The Trick-07 and Trick-10 simulation engines satisfies these constraints.
187  *
188  * \par Assumptions and Constraints on the Simulation Developer
189  * This class places certain limitations on the architecture of a JEOD-based
190  * simulation.
191  * - The JeodMemoryManager destructor uses the simulation's message handler
192  * to report errors discovered during destruction and may eventually use
193  * the simulation's simulation engine memory interface to revoke the
194  * registration of memory allocated by JEOD that has not been freed.
195  * This in turn means that:
196  * -- The simulation's message handler and simulation engine memory
197  * interface must be destructed after destructing the memory manager.
198  * -- The destructors for those objects cannot use the memory manager.
199  * - The JEOD memory allocation and deallocation macros expand into calls
200  * to memory manager methods. The memory manager must be viable (post
201  * construction, pre destruction) for these calls to function properly.
202  * This in turn means that the memory manager must be constructed very
203  * early in the overall construction process and destructed very late in
204  * in the overall destruction process.
205  * - The supported solution to both of these issues is to use a compliant
206  * derived class of the JeodSimulationInterface class and to ensure that
207  * this composite object created early and destroyed late. In a Trick-07
208  * simulation, this can be accomplished simply by placing a declaration of
209  * an object of type JeodTrickSimInterface near the top of an S_define file.
210  * The recommended placement is just after the Trick system sim object.
211  */
