101 /**
102  * This class provides the interface between the macros in jeod_alloc.hh and
103  * the rest of the JEOD memory model.
104  * The public interface is via the publicly visible static methods.
105  * All nonstatic member functions are private.
106  * Each public static method relays the method call to the singleton memory
107  * manager via a correspondingly named private member function.
108  *
109  * \par Singleton
110  * The class is intended to be a singleton.
111  * The private static member JeodMemoryManager::Master points to this
112  * singular instance. The constructor sets that static member if it is null.
113  * The constructor issues a non-fatal error when multiple instances of the
114  * class are created.
115  *
116  * \par Thread Safety
117  * This class contains objects that must be accessed and updated in a
118  * thread-safe manner. The member data that must be used atomically are
119  * - JeodMemoryManager::alloc_table - Maps memory addresses to memory items
120  * - JeodMemoryManager::type_table - Maps RTTI names to type descriptors
121  * - JeodMemoryManager::string_table - Maps unique strings to themselves.
122  * - JeodMemoryManager::cur_data_size - Current size of allocated data.
123  * - JeodMemoryManager::max_data_size - Maximum of the above.
124  * - JeodMemoryManager::max_table_size - Maximum allocation table size.
125  * - JeodMemoryManager::allocation_number - Number of allocations made.
126  * \par
127  * To ensure the constraint is satisfied, access to the these elements is
128  * protected by means of a mutex and is limited to a small number of methods.
129  * A pair of methods, JeodMemoryManager::begin_atomic_block and
130  * JeodMemoryManager::end_atomic_block systematize the use of the mutex.
131  * The methods that operate on the protected data are
132  * - Constructor and destructor.@n
133  * The constructor operates on the protected data before it creates the
134  * mutex and marks the JeodMemoryManager object as usable. The destructor
135  * marks the object as unusable and destroys the mutex before operating
136  * on the protected data.
137  * - JeodMemoryManager::generate_shutdown_report, which is called by the
138  * destructor after it has destroyed the mutex.
139  * - Methods whose names end with _atomic. These methods use the
140  * begin_atomic_block / end_atomic_block paradigm to ensure that the
141  * operations are carried out atomically.
142  * - Methods whose names end with _nolock. These methods operate on protected
143  * data but do so without atomic protection. These methods are called
144  * only by _atomic methods from within their atomic protection block.
145  *
146  * \par Forbidden Word - Mutable
147  * The data member JeodMemoryManager::mutex is mutable, a forbidden word
148  * per the JEOD coding standards. The coding standards allow for waivers to
149  * the standards if the exception is justified. This section provides the
150  * explanation needed to enable the use of that word in this case.
151  * \par
152  * The @a mutable keyword tells the compiler to ignore modifications to
153  * mutable elements in an otherwise @a const method. The @a mutex is mutable
154  * because, athough its value does change with a successful lock, it is restored
155  * to its prelock value with an unlock. A method that could otherwise qualify
156  * as a const method can still be a const method by marking the mutex as
157  * mutable. Mutexes are one of the well-accepted types of data that typically
158  * marked as mutable.
159  *
160  * \par Assumptions and Constraints on the Simulation Developer
161  * This class places restrictions on the simulation developer.
162  * - The simulation's MessageHandler object must be constructed prior
163  * to constructing the simulation's JeodMemoryManager object.
164  * - The simulation's MessageHandler object must not be destroyed prior
165  * to constructing the simulation's JeodMemoryManager object.
166  * - The simulation's JeodMemoryManager object must be constructed prior to
167  * invoking any of the JEOD_ALLOC_xxx macros in other models.
168  * - The simulation's JeodMemoryManager object must not be destroyed before
169  * other models release their allocated memory.
170  * \par
171  * The recommended solution is to create an instance of a compliant
172  * SimulationInterface before creating any other models and to destroy
173  * that SimulationInterface object after destroying all other models.
174  * A simple way to achieve this in a Trick-07 simulation is to define a
175  * Trick sim object that contains a TrickSimulationInterface element and
176  * to place this sim object immediately after the sys sim object.
177  *
178  * \par Assumptions and Constraints on the Simulation Engine
179  * This class makes certain assumptions of the behavior of the simulation
180  * engine.
181  * - The simulation engine will not spawn threads that use the JEOD memory
182  * model to allocate memory until after the SimulationInterface object has
183  * been constructed.
184  * - The simulation engine will join all threads that use the JEOD memory
185  * model prior to destroying the SimulationInterface object.
186  * \par
187  * The Trick-07 and Trick-10 simulation engines satisfies these constraints.
188  *
189  * \par Assumptions and Constraints on the Simulation Developer
190  * This class places certain limitations on the architecture of a JEOD-based
191  * simulation.
192  * - The JeodMemoryManager destructor uses the simulation's message handler
193  * to report errors discovered during destruction and may eventually use
194  * the simulation's simulation engine memory interface to revoke the
195  * registration of memory allocated by JEOD that has not been freed.
196  * This in turn means that:
197  * -- The simulation's message handler and simulation engine memory
198  * interface must be destructed after destructing the memory manager.
199  * -- The destructors for those objects cannot use the memory manager.
200  * - The JEOD memory allocation and deallocation macros expand into calls
201  * to memory manager methods. The memory manager must be viable (post
202  * construction, pre destruction) for these calls to function properly.
203  * This in turn means that the memory manager must be constructed very
204  * early in the overall construction process and destructed very late in
205  * in the overall destruction process.
206  * - The supported solution to both of these issues is to use a compliant
207  * derived class of the JeodSimulationInterface class and to ensure that
208  * this composite object created early and destroyed late. In a Trick-07
209  * simulation, this can be accomplished simply by placing a declaration of
210  * an object of type JeodTrickSimInterface near the top of an S_define file.
211  * The recommended placement is just after the Trick system sim object.
212  */
